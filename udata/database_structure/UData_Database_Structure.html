    • Security Model
      • Description
        • The security model uses the concept of multitiered permission sets.
          The main login information is first collected in 'user_accts_holding'
          at first account setup. This information is used to send and then verify
          the users account request through the email they provided. Once verified,
          the information is transferred to the 'user_accts' table for permanent storage
          and removed from the holding table. At this point the new user has no roles
          and therefore no permission sets. It is up to the administrator to set the users
          roles, which in turn have that roles permission sets. The roles are stored in the
          'roles' table. The permissions are stored in the 'perms' table. The users accounts
          are linked through relation to the roles table by the 'user_roles' table. The permissions
          are linked to the roles by relation to the 'role_perms' table. The last table is the
          'auth_tokens' table. This table stores the token given when a user successfully logs
          in and requests to remain logged in. This token is stored on the user machine and
          used to verify the user on subsequent accesses.

    • File Handling
      • Description
        • All files are indexed on the hard disk according to the scheme in the 'file_index' table.
          The individual catergories of files are stored in there own table, Safety Data Sheets = 'sds', Safety Board Meeting Minutes = 'sbm', Videos and pictures = 'media'. These tables
          store the files information categorically, however the 'file_index' stores the file information
          realistically. When a file is addressed it is address by it's id in the file_index.

    • Profiling
      • Description
        • All profiling is centralized as well in the 'profiles' table. This makes the data
          search results quicker by centralizing the search index. Anyone can be profiled
          with this method. Additional quantifying is possible at a later date, without reorganization
          of data or it's structure.

    • Products
      • Description
        • The production schema has been rendered more dis-ambiguous with the capability
          to keep and generate a running inventory. The work station has been replaced with the
          work cell, and parts are keyed by product and not to station/cell. This makes the work cell
          merely a category for parts as opposed to a reference for parts.
